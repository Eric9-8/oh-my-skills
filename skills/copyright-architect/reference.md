# 软件著作权申请材料填写规范

> 本文件是 copyright-architect skill 的填写规范参考，供生成材料时遵循。

---

## 一、软件全称命名规范

### 格式
```
[品牌/公司名] + [产品用途] + "软件"
```

### 示例
| 正确 | 错误 |
|------|------|
| XX智能客服管理软件 | XX客服 |
| XX数据分析平台软件 | 数据分析系统 |
| XX医疗影像处理软件 | 医疗影像处理 |
| XX工业设备监控软件 | 设备监控平台 |

### 注意事项
- 名称中必须包含"软件"二字
- 名称应体现软件的核心功能和用途
- 避免使用过于宽泛的描述（如"管理软件"）
- 品牌名可以是公司名、产品品牌名或个人名

---

## 二、版本号规范

- 首次申请：V1.0
- 格式：V主版本号.次版本号（如：V1.0、V2.1、V3.0）
- 不使用：1.0.0、v1.0、Version 1.0

## 三、功能描述语言规范

### 主要功能描述（≥100字）

**写作要点**：
1. 说明软件解决的核心问题
2. 列举主要功能模块（3-5个）
3. 描述软件的应用场景
4. 说明目标用户群体

**示例结构**：
```
本软件是一款[面向领域]的[软件类型]，主要用于[核心用途]。
软件提供以下核心功能：
（1）[功能1]：[功能描述]；
（2）[功能2]：[功能描述]；
（3）[功能3]：[功能描述]；
（4）[功能4]：[功能描述]。
本软件适用于[目标用户]，能够有效解决[核心问题]，提升[效率/质量/安全性等]。
```

### 技术特点描述

**写作要点**：
1. 突出技术创新点
2. 说明采用的关键技术
3. 描述性能优势
4. 避免使用营销语言

**示例**：
- 采用[算法名称]算法，实现[功能]，处理效率提升[X]%
- 基于[技术框架]构建，支持[并发数]并发访问
- 采用[加密方式]对数据进行加密存储，确保数据安全
- 实现[技术特性]，支持[功能特性]

## 四、运行环境描述规范

### 操作系统描述
- Windows：Windows 10 专业版及以上 / Windows Server 2019及以上
- Linux：Ubuntu 20.04 LTS及以上 / CentOS 7及以上 / Debian 11及以上
- macOS：macOS 12 Monterey及以上
- 移动端：Android 10及以上 / iOS 14及以上

### 硬件环境描述
- 处理器：x86-64架构，主频2.0GHz以上
- 内存：8GB以上（推荐16GB）
- 硬盘：100GB以上可用空间（SSD推荐）
- 网络：100Mbps以上带宽（如需联网）

### 支撑软件描述
- 数据库：MySQL 8.0 / PostgreSQL 15 / MongoDB 6.0
- 缓存：Redis 7.0
- Web服务器：Nginx 1.24 / Apache 2.4
- 运行时：Python 3.11 / JDK 17 / Node.js 18 / .NET 7.0

## 五、AI辅助开发声明规范（2026年3月15日新规）

### 声明原则
- 如实声明，不得虚报
- 虚假声明将影响著作权的有效性
- 使用AI工具不影响著作权申请，但必须声明

### 常见AI工具
| 工具名称 | 主要用途 |
|---------|---------|
| GitHub Copilot | 代码补全、代码生成 |
| ChatGPT / GPT-4 | 代码生成、文档编写、问题解答 |
| Claude | 代码生成、文档编写、代码审查 |
| Cursor | 代码补全、代码生成、重构 |
| Codeium | 代码补全 |
| Tabnine | 代码补全 |
| 通义灵码 | 代码补全、代码生成 |
| 文心一言 | 文档编写、代码生成 |

### 人工创作比例估算指导
- 90%以上：AI仅用于代码补全，核心逻辑完全由人工编写
- 70-90%：AI辅助生成部分代码框架，核心业务逻辑由人工实现
- 50-70%：AI生成较多代码，人工进行修改、调试和业务逻辑实现
- 50%以下：大量使用AI生成代码，需特别说明人工创作的独特贡献

## 六、开源组件声明规范（2026年审查要求）

### 声明原则
- 使用开源组件必须声明
- 纯复制开源代码不予登记，需有二次开发
- 说明二次开发/修改/封装的具体内容

### 常见许可证类型
| 许可证 | 特点 | 商业使用 |
|--------|------|---------|
| MIT | 最宽松，可商用 | 允许 |
| Apache 2.0 | 宽松，可商用，需保留版权声明 | 允许 |
| BSD | 宽松，可商用 | 允许 |
| GPL v2/v3 | 传染性，修改需开源 | 受限 |
| LGPL | 动态链接可商用 | 受限 |
| AGPL | 网络服务也需开源 | 受限 |

## 七、软件操作说明书写作规范

### 总体要求
- 总字数不少于3000字
- 每个功能模块描述不少于150字
- 图文结合，截图需配有文字说明
- 语言准确、专业，避免口语化表达
- 技术术语使用一致

### 一致性要求（2026年审查红线）
- 说明书中的软件全称必须与申请表完全一致
- 版本号必须与申请表完全一致
- 功能描述的核心内容必须与申请表一致
- 不一致将导致补正或驳回

### 截图规范
- 截图需清晰，分辨率不低于72dpi
- 截图中的软件界面需显示软件名称或版本信息
- 截图需配有文字说明，描述界面功能
- 如无法提供真实截图，可使用界面示意图并注明

### 流程图规范
- 使用标准流程图符号（开始/结束用椭圆，判断用菱形，处理用矩形）
- 标注数据流向（箭头方向）
- 标注关键决策节点的判断条件
- 每个流程图需配有文字说明

## 八、2026年审查重点速查

### 审查红线（必须避免）
1. ❌ 文档/源代码/申请表名称版本不一致
2. ❌ 截图、流程、逻辑虚构或不可核验
3. ❌ 仅放截图无文字、无流程、无功能说明
4. ❌ 通用模板拼接，无独创性表达
5. ❌ 纯复制开源代码未声明
6. ❌ 注释/空行凑数（有效代码<50行/页）
7. ❌ AI相似度过高（与版权库/GitHub/Gitee比对）

### 审查加分项
1. ✅ 文档结构完整，章节清晰
2. ✅ 图文结合，截图真实
3. ✅ 技术描述具体，体现独创性
4. ✅ AI辅助声明如实填写
5. ✅ 开源组件声明完整，说明二次开发内容
6. ✅ 核心代码有清晰注释
7. ✅ 测试用例真实，有测试结果
